Bug Hunt branch: main-errors

19 intentional bugs planted across the Banking System API. Find them all.

Logic Auth / Roles SQL / Data HTTP Response Routing
01 Logic

GET client by ID returns the wrong client

GET /api/v1/clients/{id}

When id = 1 the correct client is returned. For any other ID, the API silently looks up id - 1 instead, returning a different client's data.

routes/clients.py → ClientDetail.get()
02 SQL / Data

Client accounts endpoint always returns accounts for client 1

GET /api/v1/clients/{id}/accounts

Regardless of which client ID is passed, the SQL query is hardcoded to ClientID = 1. Every client appears to own the same accounts.

routes/clients.py → ClientAccounts.get()
03 Logic

Client summary FullName is reversed (LastName + FirstName)

GET /api/v1/clients/{id}/summary

The SQL concatenation is LastName + ' ' + FirstName instead of FirstName + ' ' + LastName, so every full name comes back surname-first.

routes/clients.py → ClientSummary.get()
04 Logic

New client is stored with FirstName and LastName swapped

POST /api/v1/clients/

The INSERT passes p['LastName'] into the FirstName column and p['FirstName'] into LastName. Names are silently flipped in the database.

routes/clients.py → ClientList.post()
05 SQL / Data

Client search filter is completely ignored

GET /api/v1/clients/?search=...

The search query parameter is parsed but never used. The query always returns all clients, so filtering by name or email has no effect.

routes/clients.py → ClientList.get()
06 Auth / Roles

Creating a bank account requires readonly role, not admin

POST /api/v1/accounts/

The decorator is @require_auth(roles=['readonly']). Admin tokens receive a 403, while readonly tokens — which should have no write access — can create accounts.

routes/accounts.py → AccountList.post()
07 HTTP Response

GET account by ID always returns 200 with empty body

GET /api/v1/accounts/{id}

The handler immediately returns {} without querying the database. All IDs — including non-existent ones — return 200 with an empty object instead of actual data or a 404.

routes/accounts.py → AccountDetail.get()
08 SQL / Data

Account transactions always returns transactions for account 1

GET /api/v1/accounts/{id}/transactions

The SQL is hardcoded to AccountID = 1. Every account returns the same transaction history regardless of the requested account ID.

routes/accounts.py → AccountTransactions.get()
09 Logic

Deposit subtracts from balance instead of adding

POST /api/v1/transactions/deposit

balance_after is calculated as balance_before - amount instead of balance_before + amount. Every deposit reduces the account balance.

routes/transactions.py → Deposit.post()
10 Logic

Withdrawal allows overdraft — balance check is broken

POST /api/v1/transactions/withdrawal

The guard condition is if balance_before < 0 instead of if balance_before < amount. Accounts can be overdrawn to any negative value without error.

routes/transactions.py → Withdrawal.post()
11 SQL / Data

GET transaction by ID always returns the oldest transaction

GET /api/v1/transactions/{id}

The query ignores transaction_id and instead runs SELECT TOP 1 * FROM Transactions ORDER BY TransactionID ASC, always returning the very first transaction ever created.

routes/transactions.py → TransactionDetail.get()
12 Logic

Transaction list is sorted oldest-first instead of newest-first

GET /api/v1/transactions/

The ORDER BY clause uses ASC instead of DESC, so the most recent transactions appear at the end of the list, not the beginning.

routes/transactions.py → TransactionList.get()
13 Routing

Transfer endpoint URL is misspelled

POST /api/v1/transactions/transfurer

The route is registered as /transfurer instead of /transfer. Calls to the correct URL return 404; the endpoint only responds on the misspelled path.

routes/transactions.py → Transfer class
14 SQL / Data

GET loan by ID looks up the wrong loan (id - 1)

GET /api/v1/loans/{id}

The query uses LoanID = loan_id - 1, so every request returns the previous loan. Loan 5 returns loan 4's data; loan 1 returns nothing (404).

routes/loans.py → LoanDetail.get()
15 SQL / Data

Loan payments return payments for loan_id + 1

GET /api/v1/loans/{id}/payments

The payment query uses LoanID = loan_id + 1, returning the next loan's payments. The last loan in the system returns an empty list.

routes/loans.py → LoanPayments.get()
16 HTTP Response

Create branch returns 200 instead of 201

POST /api/v1/branches/

The @ns.marshal_with decorator is configured with code=200 instead of code=201. A successful creation returns 200 OK instead of the correct 201 Created.

routes/branches.py → BranchList.post()
17 Auth / Roles

GET branch by ID requires no authentication

GET /api/v1/branches/{id}

The @require_auth() decorator was removed from this endpoint. Anyone — with or without a token — can retrieve branch details without authenticating.

routes/branches.py → BranchDetail.get()
18 SQL / Data

Cards by client always returns cards for client 1

GET /api/v1/cards/client/{id}

The SQL join is hardcoded to a.ClientID = 1. All clients appear to have the same cards — those belonging to client 1.

routes/cards.py → CardsByClient.get()
19 Logic

Exchange rate pair lookup swaps base and target currencies

GET /api/v1/exchange-rates/{base}/{target}

The query passes (target, base) instead of (base, target). Requesting USD → ILS actually returns the ILS → USD rate, giving the inverse exchange rate.

routes/exchange_rates.py → ExchangeRatePair.get()
20 Logic

Loan application decision always approves, regardless of input

POST /api/v1/loan-applications/{id}/decision

The variable decision is hardcoded to 'approved' before the payload is read. Submitting "decision": "rejected" still marks the application as approved.

routes/loan_applications.py → LoanApplicationDecision.post()
21 SQL / Data

Loan application status filter is completely ignored

GET /api/v1/loan-applications/?status=...

The status query parameter is parsed but the conditional branch is removed — all applications are always returned, making status filtering non-functional.

routes/loan_applications.py → LoanApplicationList.get()
22 Logic

New loan application is always filed under client 1

POST /api/v1/loan-applications/

After validating the provided ClientID, the code overwrites it with p['ClientID'] = 1. Every application is stored against client 1 regardless of which client was specified.

routes/loan_applications.py → LoanApplicationList.post()
